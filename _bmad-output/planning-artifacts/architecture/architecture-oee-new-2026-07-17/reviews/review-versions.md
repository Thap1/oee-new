# Version/Technology Reality-Check — ARCHITECTURE-SPINE.md

Scope: every version/technology claim in the Stack table plus every named
technology/pattern elsewhere in the document (SignalR, JWT/JWKS, GUID v7,
EF Core). Each item was checked with live web search against today's date
(2026-07-17), not answered from model training data.

## Verdict

Most core version numbers in the Stack table check out and are accurate as
of today, but two fit-for-purpose claims do not survive scrutiny: **PrimeNG
22.0.0 quietly ships a new non-MIT dual-license model that the doc never
mentions**, and **GUID v7 (AD-6) as generated by native .NET tooling has a
documented ordering defect that defeats the index-fragmentation benefit v7
is normally chosen for on Postgres** — both are "committed decisions" that
need a follow-up note in the spine, not silent acceptance.

## Findings

### 1. [HIGH] PrimeNG 22.0.0 — undisclosed licensing model change, Angular-22 support unconfirmed at commit time

The doc's own text ("PrimeNG vừa chuyển sang tổ chức PrimeUI, repo cũ đã
archive") is directionally correct but incomplete in a way that matters for
a "committed" stack decision:

- `primefaces/primeng` (the old repo) **was archived on 2026-06-28** and is
  read-only. Confirmed via the repo's own archive banner language ("This
  repository is now archived... PrimeNG enters its next chapter as part of
  PrimeUI").
- As late as **2026-06-04** (GitHub issue `primefaces/primeng#19608`,
  "Angular 22 Support"), PrimeNG only officially supported **Angular 21**,
  even though Angular 22 had already shipped (2026-06-03). The issue closed
  with no confirmed resolution before the repo went read-only.
- **PrimeNG v22.0.0 is the first release published under the new PrimeUI
  org**, and per its own migration guide: *"PrimeNG v22 is the first release
  under the PrimeUI umbrella. Licensing moves to a dual Community/Commercial
  model."* This is a **material change from the MIT license** every prior
  PrimeNG version (≤21) shipped under.
  - Community License (free) requires an org to have **<$1M annual
    revenue, <5 developers, <10 total employees, and <$3M in outside
    funding raised, ever** — government, public-sector, and university
    orgs are excluded outright regardless of size.
  - Orgs that don't qualify must buy the **Commercial Suite**
    ($599/developer perpetual through 2026, rising to $799/developer
    after) — a real procurement cost this architecture doc doesn't
    surface anywhere.
  - Staying on PrimeNG ≤21 remains free/MIT forever, but (per the
    unresolved GitHub issue above) v21 has **no confirmed Angular 22
    support**, which conflicts with the doc's own choice of Angular 22.0.
- I could not independently confirm from npm (403 on fetch) that
  `primeng@22.0.0`'s `peerDependencies` actually declare Angular 22 support.
  Given v22.0.0 was published essentially the same week this architecture
  doc was written, there has been **~0 field-testing time** on this
  exact pairing.

**Why it matters for this project:** the doc treats "PrimeNG 22.0.0" as a
scaffold-verified fact (package exists, name confirmed) but doesn't carry
forward the licensing consequence, which is a **procurement/legal decision**,
not just a technical one — the project needs to either (a) confirm it
qualifies for the Community License, (b) budget for Commercial, or (c) pin
to PrimeNG 21 + Angular 21 instead of 22 until compatibility is proven out.
This should be promoted from "Stack table footnote" to an explicit
Invariant or Deferred-with-owner item.

- https://github.com/primefaces/primeng/issues/19608 (Angular 22 Support, opened 2026-06-04, unresolved)
- https://primeng.dev/migration/v22 (dual Community/Commercial licensing language)
- https://primeui.dev/nextchapter (transition FAQ: MIT versions "not retroactive", v22+ requires license)
- https://primeui.dev/licenses/community (eligibility thresholds)

### 2. [HIGH] GUID v7 (AD-6) — collision-avoidance claim holds, but the implied index-locality benefit does not, on the stated stack

AD-6 justifies GUID v7 as the identity strategy for all synced entities to
avoid PK collisions when two independent sites both mint IDs before syncing.
That collision-avoidance property is real and correct regardless of
implementation (128-bit space, effectively zero collision probability for
this data volume).

However, the natural reason to pick **v7 over v4** specifically (rather
than "any GUID") is v7's time-ordered prefix, which is supposed to produce
smaller, less-fragmented B-tree indexes than random v4 — directly relevant
since the stack is Postgres. That benefit **does not automatically hold**
if IDs are generated the "obvious" way in .NET application code:

- `Guid.CreateVersion7()` (introduced .NET 9, carried into .NET 10) has a
  **confirmed, currently-unfixed** ordering defect: it stores the 48-bit
  timestamp in the GUID's internal byte layout in a way that does not match
  RFC 9562's required big-endian byte order for storage/sorting purposes
  (it only "looks" correct in the string representation). Because Postgres
  compares `uuid` values byte-for-byte, this causes the effective sort key
  to **rotate through byte values roughly every few hours instead of
  monotonically increasing** — reintroducing the same random-insert B-tree
  fragmentation that v7 was chosen to prevent. This is documented
  independently against Postgres specifically (not just SQL Server) in
  `dotnet/runtime#111503`.
- Separately, `Guid.CreateVersion7()` fills everything after the timestamp
  with pure randomness (no monotonic counter per RFC 9562 §6.2), so IDs
  generated within the same millisecond are unordered relative to each
  other — a second, independent source of index fragmentation under
  concurrent inserts (which this system will have, given multiple machines
  reporting ingestion events concurrently at each site).
- The good news: **PostgreSQL 18 ships a native, correct `uuidv7()`
  SQL function** (core, no extension) that guarantees monotonic ordering
  within a backend session and gets the documented ~26–27% smaller
  index / ~3x faster ordered-scan benefit in published benchmarks. Since
  the stack already commits to Postgres 18, the fix is straightforward:
  generate the ID via `DEFAULT uuidv7()` (or an EF Core value generator
  that calls it / a corrected client library such as `Medo.Uuid7`) rather
  than calling `Guid.CreateVersion7()` directly in application code.

**Recommendation:** AD-6's rule should be tightened from "dùng Guid v7 hoặc
v4" to something that explicitly rules out naive `Guid.CreateVersion7()`
for anything hitting a Postgres PK/index — either delegate generation to
Postgres's native `uuidv7()`, or explicitly pick v4 if app-side generation
in .NET is required and index locality isn't actually needed for that
entity.

Sources:
- https://github.com/dotnet/runtime/issues/111503 (byte-order defect, explicitly confirmed against Postgres, "closed" but unresolved as a real defect)
- https://gist.github.com/sdrapkin/03b13a9f7ba80afe62c3308b91c943ed ("Avoid using Guid.CreateVersion7 in .NET")
- https://github.com/dotnet/SqlClient/issues/2989 (parallel SQL Server-side report of the same root cause)
- https://www.postgresql.org/docs/18/release-18-4.html and general PG18 release notes (native `uuidv7()`)
- https://www.thenile.dev/blog/uuidv7 and https://betterstack.com/community/guides/databases/postgresql-18-uuid/ (PG18 native uuidv7, monotonicity guarantee, benchmark numbers)

### 3. [MEDIUM] JWT/JWKS offline-first pattern (AD-7) — sound pattern, but rotation/grace-period policy is left undefined despite being load-bearing for the offline story

Industry practice confirms the doc's approach is directionally correct:
caching JWKS locally and validating signatures without a round trip to the
issuer is the standard pattern (RFC 7517 JWKS, RFC 8414/8615 for the
`.well-known` discovery convention), and "cached keysets for
air-gapped/intermittently-connected services" paired with alerting on
drift is explicitly called out as acceptable practice.

The gap: the doc explicitly defers "Cơ chế rotation/refresh chính xác của
JWKS tại site" as "implementation detail, không phải invariant" (bottom of
the Deferred section). But this system's whole selling point for AD-7 is
that a site can run disconnected from Central for extended periods (same
premise as AD-2's "site tự trị"). If Central rotates its signing key while
a site is cut off, and that site's cached JWKS never refreshes, then either
(a) new tokens issued during/after rotation fail to validate at the
disconnected site once it reconnects and the site doesn't re-fetch, or (b)
if the site refreshes proactively, there's no invariant guaranteeing this
happens before the old key is retired centrally. Given the multi-site,
long-disconnection design is a first-class scenario here (not an edge
case), this arguably **should** be promoted from "Deferred implementation
detail" to at least a stated invariant of the form "sites must refresh
JWKS at least every N hours when connected, and Central must keep retired
keys valid in JWKS for at least 2×N hours after rotation" — otherwise the
offline-first authentication story has an unstated failure mode.

This is not a version/existence problem (JWT/JWKS is real, standard, and
correctly described) — it's a fit-for-this-architecture gap around a
detail the doc chose to wave off as non-architectural, when in this
specific multi-site-offline design it arguably is architectural.

Sources:
- https://www.davidsulc.com/blog/jws-apis-jwks-basics (JWKS rotation mechanics)
- https://nhimg.org/faq/what-should-teams-do-when-jwks-keys-rotate-frequently/ (cached keysets for air-gapped/intermittent services, paired with alerting)
- https://curity.io/resources/learn/token-signing-key-rotation/ (grace period / overlapping key validity pattern)

## Net Recommendation

1. Add an explicit note/invariant addressing PrimeNG's new licensing model
   (Community-license eligibility check or Commercial budget line, or a
   fallback pin to PrimeNG 21/Angular 21) before this becomes a blocking
   surprise during implementation.
2. Tighten AD-6 to specify *how* GUID v7 values are generated (Postgres
   native `uuidv7()` or a corrected library) rather than leaving
   `Guid.CreateVersion7()` as an implicit default, since the naive path
   silently forfeits the reason v7 was chosen over v4.
3. Consider promoting JWKS rotation/grace-period policy from "Deferred
   implementation detail" to a stated invariant, given that long
   site-disconnection is core to this architecture's design, not an edge
   case.
