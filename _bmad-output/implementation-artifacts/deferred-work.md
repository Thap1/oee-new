## Deferred from: code review of 1-4-user-role-scope-management (2026-07-24)

- **AD-7's site-local-scoping vs. central-credential split isn't actually implemented (both live on the same `User` row/DB).** Investigated 2026-07-24 at the user's request to implement it for real — confirmed blocked on two separate pieces of infrastructure that don't exist yet, not fixable as a single-file patch: (1) `SyncBatch` (`src/OeeNew.Application/Sync/SyncBatch.cs`) has no User/role-scoping record type at all — there is no wire path for a Site to push role-scoping to Central; (2) `RsaJwtSigningKeyProvider` (`src/OeeNew.Infrastructure/Identity/RsaJwtSigningKeyProvider.cs`) is a per-process, in-memory-only RSA key store (`AddSingleton`) with no cross-instance JWKS federation — a JWT signed by one instance can never validate against another. A real implementation needs both solved first (extend the sync payload + build key/JWKS federation across instances), which is Epic-level scope, not a review patch. **Recommendation: scope as its own story/epic (e.g. "Cross-Instance Identity Federation") once Epic 5's sync infrastructure is otherwise stable**, rather than bolting it onto Story 1.4. [src/OeeNew.Domain/Identity/User.cs, src/OeeNew.Application/Sync/SyncBatch.cs, src/OeeNew.Infrastructure/Identity/RsaJwtSigningKeyProvider.cs]
- No optimistic concurrency token on `User` updates — last-write-wins on concurrent `PUT`. Consistent with every other master-data entity in the codebase today, not a regression specific to this story. [src/OeeNew.Infrastructure/Persistence/UserRepository.cs:25-26]
